SMS Notifications (provider-agnostic) — BHELA Booking

Mirror to wp-content/docs/plans/2026-07-14-sms.md on implementation (project rule).

Context

Owner wants an SMS on every new booking (to owner + to customer) and on status changes (to customer), but hasn't picked a gateway ("confused to choose provider"). So build a configurable HTTP SMS sender inside the existing bhela-booking plugin: works with any Bangladesh gateway (BulkSMSBD, Alpha, MIM, Reve, SSL, etc.) via admin settings — no code edits to switch providers. Ships a BulkSMSBD preset (most common) plus a fully-custom mode. Everything hangs off the notification points the plugin already uses for email, and the customer phone is already stored as _bhela_phone.

New file: plugins/bhela-booking/includes/sms.php (required from bhela-booking.php)

Core sender bhela_bm_send_sms( $number, $message ):

  • Returns early (false) unless sms_enabled.
  • Normalize BD number → 88 + 11-digit (strip non-digits, add 88 if missing).
  • Build request by provider:
  • bulksmsbd: GET https://bulksmsbd.net/api/smsapi?api_key={key}&senderid={sid}&number={to}&message={msg}.
  • custom: admin supplies base URL, method (GET/POST), and param names for number/message; api key + sender id sent as their configured param names; optional Authorization header. Body form-encoded (or JSON if a "json" toggle set).
  • wp_remote_get/post with 15s timeout; capture HTTP code + body into bhela_bm_sms_last option (masked — never store full key) for an admin status line. Return bool.
  • Helper bhela_bm_render_sms( $template, $booking_id ) replaces placeholders: {name} {invoice} {date} {cabin} {guests} {total} {advance} {due} {status} using existing meta + bhela_bm_money() + bhela_bm_statuses().

Settings — extend bhela_bm_default_settings() (bhela-booking.php) with: sms_enabled(0), sms_provider('bulksmsbd'|'custom'), sms_api_url, sms_method('GET'), sms_json(0), sms_api_key, sms_sender_id, sms_param_number('number'), sms_param_message('message'), sms_param_key('api_key'), sms_param_sender('senderid'), sms_auth_header(''), sms_admin_number(defaults to phone_1), and 3 templates: sms_tpl_admin, sms_tpl_new, sms_tpl_confirmed with sensible Bangla defaults.

Add an "📱 SMS Notifications" section to the settings page (admin.php bhela_bm_settings_page, mirroring the existing sanitized POST loop at admin.php:422-446): enable toggle, provider dropdown, credentials, param mapping (shown for custom), admin number, 3 template textareas with the placeholder legend, and a "Send Test SMS" button (its own nonce'd admin_post action → sends the admin template to sms_admin_number, prints the gateway's response).

Hook the 3 triggers (reuse existing send points)

  1. New booking → customer and → admin: in bhela_bm_process_submission() right after the email calls (frontend.php ~430-432) —

bhela_bm_send_sms( $phone, bhela_bm_render_sms( $s['sms_tpl_new'], $post_id ) ) and bhela_bm_send_sms( $s['sms_admin_number'], bhela_bm_render_sms( $s['sms_tpl_admin'], $post_id ) ).

  1. Status change → customer: in bhela_bm_save_booking() (admin.php:376-381), on any real transition $new_status !== $old_status, send sms_tpl_confirmed (template text adapts via {status} label). Also fire from the front-end status flow if one exists.

All sends wrapped so an SMS failure never blocks the booking/email (log + continue).

Security

  • Settings + test: current_user_can('manage_options') + nonce (check_admin_referer), sanitize_text_field/esc_url_raw/sanitize_textarea_field, absint toggles.
  • API key stored in option (same trust level as bKash numbers already there); never echoed back in full (show ••••last4), never logged.
  • Outbound message via rawurlencode; number regex-cleaned.
  • No unauthenticated trigger — SMS only fires from server-side booking/save paths.

Version + docs

Verification (SMS can't actually deliver on Local)

  1. Payload correctness without a real gateway: point sms_api_url at a local capture — a tiny wp_remote_* intercept via the pre_http_request filter in a throwaway mu-test — assert the built URL/body contains the normalized 8801… number and rendered message. Confirm GET vs POST + param names honoured for both bulksmsbd and custom.
  2. Triggers fire: submit a booking (curl, valid nonce) with SMS enabled + intercept active → assert two intercepted sends (customer + admin) with correct templates/placeholders resolved.
  3. Status change: flip a booking pending→confirmed via the admin save path (CLI bhela_bm_save_booking simulation) → assert one customer SMS with {status} = "Confirmed (নিশ্চিত)".
  4. Disabled = silent: sms_enabled=0 → intercept sees zero calls; booking + email still succeed.
  5. Number normalization: 01712…, +8801712…, 8801712… all → 8801712….
  6. Settings UI: save section, "Send Test SMS" returns the gateway response line; key shown masked. Regression: booking email flow unaffected (Mailpit).
